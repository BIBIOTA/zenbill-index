# ZenBill 分類重整設計

**日期:** 2026-02-23
**狀態:** 已批准

## 背景

目前 ZenBill 有 57 個分類（50 EXPENSE + 5 INCOME + 2 子分類），幾乎全部扁平。存在語意重複、粒度不一致、缺乏層級結構等問題。

### 現有問題

1. **語意重複** — 食品 vs 餐飲、3C vs 電子產品、手續費 vs 信用卡交易手續費
2. **粒度不一致** — 大分類（娛樂）與細分類（KTV、路跑）同層
3. **缺乏層級** — 加油/停車/計程車 都跟交通相關但各自獨立
4. **14 個空分類** — 0 筆交易的分類佔 25%

## 設計方案

### 實作方式

純資料庫 migration script，不需修改後端程式碼（domain 已支援 parent/child 層級）。

#### 新建父分類

餐飲、交通、旅遊、娛樂、購物、訂閱服務、生活、金融、投資(EXPENSE)、薪資收入、投資收入、回饋

#### 移動至子分類（設定 parent_id）

所有現有分類根據上方結構表移入對應父分類。

#### 刪除

| 分類 | 原因 |
|------|------|
| （無類別）| 系統用 null 即可 |
| 雇主 | 0 筆，語意不明 |
| 家庭 | 0 筆，可歸入生活 |
| 餐飲 | 0 筆，與新父分類重名 |
| 旅遊 | 0 筆，新建父分類取代 |
| 娛樂 | 0 筆，新建父分類取代 |
| 投資 | 0 筆，新建父分類取代 |
| 生活 | 0 筆，新建父分類取代 |
| 汽車 | 0 筆，交通子分類已涵蓋 |
| 公共運輸 (舊) | 0 筆，但「自動加值」合併後保留此名 |

*注意：公共運輸的處理 — 保留現有「公共運輸」分類，將「自動加值」的 3 筆交易遷移過去，再把公共運輸移入交通父分類。*

### Merchant 關聯處理

合併/刪除的分類若被 `merchants.default_category_id` 引用，需同步遷移到新分類。

### 回滾策略

- 整個 migration 在單一 DB transaction 內執行
- 執行前 backup categories + transactions.category_id 對照

### 影響範圍

- 僅 DB 資料變更，不需修改後端/前端程式碼
- 前端已支援 tree 結構顯示
- API 已回傳 parent/children 層級
