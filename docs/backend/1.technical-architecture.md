## 1. 系統概觀 (System Overview)

ZenBill 是一個以「自動化」為核心的個人記帳系統。後端採用 **Go (Golang)** 開發，旨在處理高併發的發票同步、複雜的正規化規則運算 (Rule Engine)，以及精準的複式簿記 (Double-Entry Bookkeeping) 邏輯。

系統架構分為兩大運行實體：

1. **API Server:** 處理使用者即時請求 (CRUD)。
2. **Worker Service:** 背景執行排程任務 (發票網頁爬蟲、規則清洗、自動扣款)。

## 4. 資料庫設計策略 (Database Strategy)

### 4.1 核心原則

1. **ACID Transaction:** 涉及金流變動（如：新增支出並扣除餘額），務必使用 DB Transaction (`tx.Begin()`, `tx.Commit()`, `tx.Rollback()`)。
2. **JSONB for Raw Data:** `invoices` 表使用 `raw_details JSONB` 欄位儲存 API 回傳的完整明細，保留原始資料以備未來分析，不需過度正規化。
3. **Decimal Precision:** 金額欄位統一使用 `DECIMAL(19, 4)`，Go 語言層建議使用 `shopspring/decimal` 套件避免浮點數誤差。

### 4.2 關鍵 Table Schema (簡述)

* `users`: 使用者基底。
* `accounts`: 資產帳戶 (含 `current_balance` 與 `auto_pay_from_id`)。
* `transactions`: 流水帳 (Immutable 優先，修正時透過 Update 處理)。
* `invoices`: 外部資料源 (含 `status` 狀態機：PENDING -> PROCESSED)。
* `merchants` & `merchant_rules`: 正規化引擎的知識庫。

## 5. 核心模組詳細設計 (Core Modules)

### 5.1 財政部發票爬蟲模組 (`pkg/einvoice`)

* **職責:** 使用 Playwright 封裝財政部電子發票平台的爬蟲邏輯，攔截 JSON API 並提取結構化資料。
* **技術難點:**
  - **Cloudflare 防護繞過** (需等待 5-10 秒 JavaScript 挑戰完成)
  - **JSON API 攔截** (核心功能！發票資料透過 `/api/searchCarrierInvoice` 等 endpoints 以 JSON 格式返回)
  - **Session Cookie 管理** (維持登入狀態，避免每次都重新登入)
  - **OAuth 2.0 重定向處理** (login_challenge 動態 token)

* **為何選擇 playwright-go 而非 chromedp:**
  ```go
  // ✅ Playwright - 優雅地攔截 JSON API
  page.OnResponse(func(resp playwright.Response) {
      if strings.Contains(resp.URL(), "searchCarrierInvoice") {
          var invoices []Invoice
          resp.JSON(&invoices) // 一行搞定！
      }
  })

  // ❌ Chromedp - 需手動處理 CDP 事件、RequestID 對應、goroutine...
  chromedp.ListenTarget(ctx, func(ev interface{}) {
      // 複雜的底層 CDP 事件處理...
  })
  ```

* **介面:**
```go
type Scraper interface {
    // 初始化瀏覽器並登入（處理 Cloudflare + OAuth）
    Login(cardID, verifyCode string) error

    // 攔截 JSON API 並取得發票列表（核心方法）
    GetInvoices(dateRange DateRange) ([]InvoiceRaw, error)

    // 攔截 JSON API 並取得發票明細
    GetInvoiceDetail(invNum string) (InvoiceDetailRaw, error)

    // 檢查 Session 是否有效
    IsSessionValid() bool

    // 關閉瀏覽器並釋放資源
    Close() error
}
```



### 5.2 正規化規則引擎 (`internal/usecase/rule_engine.go`)

* **職責:** 將雜亂的商家名稱轉換為標準商家 ID。
* **流程:**
1. 載入該使用者的所有 Rules (依 Priority 排序)。
2. 輸入 `RawMerchantName`。
3. 迴圈比對：
* `CONTAINS`: `strings.Contains()`
* `REGEX`: `regexp.MatchString()` (建議預先 Compile Regex 以提升效能)。


4. 回傳 `TargetMerchantID` 與 `DefaultConfig`。



### 5.3 自動扣款排程 (`cmd/worker`)

* **觸發:** 每日 10:00 AM (Cron: `0 10 * * *`)。
* **邏輯:**
1. `SELECT * FROM accounts WHERE type = 'CREDIT' AND payment_due_day = today`
2. 針對每張卡，計算 `balance` (負債)。
3. 若 `balance < 0` 且有設定 `auto_pay_from_id`:
4. 開啟 Transaction:
* Insert `TRANSFER` Transaction (Bank -> Credit Card).
* Update Bank Balance (-).
* Update Credit Card Balance (+).


5. Commit.

## 6. Playwright 爬蟲實作策略 (Scraping Implementation with Playwright)

### 6.1 財政部 API 分析 (實際流程)

**登入流程:**
- **URL:** `https://www.einvoice.nat.gov.tw/accounts/login`
- **需處理:**
  - 手機號碼輸入 (例: `0917923382`)
  - 驗證碼輸入 (例: `REDACTED_VERIFICATION_CODE`)
  - 圖形驗證碼 (需人工處理或 OCR)
  - Cloudflare 挑戰 (等待 10 秒)
  - OAuth 2.0 重定向處理

**發票列表查詢:**
- **API Endpoint:** `https://service-mc.einvoice.nat.gov.tw/btc/cloud/api/btc502w/searchCarrierInvoice`
- **需攔截的欄位:**
  - `invoiceNumber`: 發票號碼 (例: "WF57443025")
  - `invoiceDate`: 發票日期 (Unix Timestamp)
  - `sellerName`: 商家名稱 (原始名稱，需正規化)
  - `totalAmount`: 總金額
  - `token`: 用於查詢明細的憑證
  - `carrierName`: 載具名稱 (例: "手機條碼")

**發票明細查詢:**
- **API Endpoint:** `https://service-mc.einvoice.nat.gov.tw/btc/cloud/api/common/getCarrierInvoiceDetail`
- **Request:** 使用發票列表中的 `token`
- **Response:**
  - `content[]`: 發票明細陣列
    - `sequenceNumber`: 序號
    - `item`: 品項名稱
    - `quantity`: 數量
    - `unitPrice`: 單價
    - `amount`: 小計

func NewScraper() (*PlaywrightScraper, error) {
    // 初始化 Playwright
    pw, err := playwright.Run()
    if err != nil {
        return nil, err
    }

    // 啟動 Chromium（Headless 模式）
    browser, err := pw.Chromium.Launch(playwright.BrowserTypeLaunchOptions{
        Headless: playwright.Bool(false), // 開發時設為 false 方便除錯
        Args: []string{
            "--disable-blink-features=AutomationControlled", // 隱藏自動化特徵
        },
    })
    if err != nil {
        return nil, err
    }

    // 建立 Browser Context（可儲存 Cookies）
    context, err := browser.NewContext()
    if err != nil {
        return nil, err
    }

    page, err := context.NewPage()
    if err != nil {
        return nil, err
    }

    return &PlaywrightScraper{
        browser: browser,
        page:    page,
        context: context,
    }, nil
}

func (s *PlaywrightScraper) Login(phoneNumber, verifyCode, captcha string) error {
    // 1. 訪問登入頁面
    if _, err := s.page.Goto("https://www.einvoice.nat.gov.tw/accounts/login"); err != nil {
        return err
    }

    // 2. 等待 Cloudflare 挑戰完成（關鍵步驟！）
    time.Sleep(10 * time.Second)

    // 3. 填寫登入表單
    if err := s.page.Fill("input[name='phoneNumber']", phoneNumber); err != nil {
        return err
    }
    if err := s.page.Fill("input[name='verifyCode']", verifyCode); err != nil {
        return err
    }

    // 4. 自動處理圖形驗證碼（使用 Tesseract OCR）
    // ✅ 已實作：使用 CLI 版本的 Tesseract 進行自動辨識
    // 位置：pkg/einvoice/captcha/
    captchaText, err := s.solveCaptcha()
    if err != nil {
        // OCR 失敗時可回退到人工輸入（Fallback 機制）
        return fmt.Errorf("驗證碼處理失敗: %w (建議手動輸入)", err)
    }

    log.Printf("✅ 驗證碼辨識成功: %s", captchaText)

    if err := s.page.Fill("input[name='captcha']", captchaText); err != nil {
        return err
    }

    // 5. 提交表單
    if err := s.page.Click("button[type='submit']"); err != nil {
        return err

        Timeout: playwright.Float(30000),

        return err
    }

    // 7. 儲存 Browser Context（包含 Cookies 和 LocalStorage）
    if err := s.saveBrowserContext(); err != nil {
        return err
    }

    return nil
}

func (s *PlaywrightScraper) GetInvoiceList(startDate, endDate time.Time) (*InvoiceListResponse, error) {
    var result *InvoiceListResponse
    responseChan := make(chan error, 1)

    // 設定 API 攔截器（核心功能！）
    s.page.OnResponse(func(resp playwright.Response) {
        if strings.Contains(resp.URL(), "searchCarrierInvoice") {
            var apiResponse InvoiceListResponse
            if err := resp.JSON(&apiResponse); err == nil {
                result = &apiResponse
                log.Printf("✅ 成功攔截到 %d 筆發票", apiResponse.TotalElements)
                responseChan <- nil
            } else {
                responseChan <- fmt.Errorf("JSON 解析失敗: %w", err)
            }
        }
    })

    // 前往查詢頁面
    if _, err := s.page.Goto("https://www.einvoice.nat.gov.tw/portal/btc/mobile/btc502w/search"); err != nil {
        return nil, err
    }

    // 填寫查詢條件（當月發票）
    startDateStr := startDate.Format("2006/01/02")
    endDateStr := endDate.Format("2006/01/02")

    if err := s.page.Fill("#startDate", startDateStr); err != nil {
        return nil, err
    }
    if err := s.page.Fill("#endDate", endDateStr); err != nil {
        return nil, err
    }

    // 點擊查詢按鈕
    if err := s.page.Click("button#search"); err != nil {
        return nil, err
    }

    // 等待 API 回應（最多 30 秒）
    select {
    case err := <-responseChan:
        if err != nil {
            return nil, err
        }
    case <-time.After(30 * time.Second):
        return nil, fmt.Errorf("API 回應逾時")
    }

    return result, nil
}

### 6.4 資料庫寫入流程 (Sync Service)

```go
// internal/usecase/sync_service.go

type SyncService struct {
    scraper      einvoice.Scraper
    invoiceRepo  domain.InvoiceRepository
}

func (s *SyncService) SyncInvoices(startDate, endDate time.Time) error {
    // 1. 登入（若 Session 無效）
    if !s.scraper.IsSessionValid() {
        // 從加密儲存中讀取憑證
        phoneNumber := s.getEncryptedCredential("phone")
        verifyCode := s.getEncryptedCredential("verify_code")

        if err := s.scraper.Login(phoneNumber, verifyCode, ""); err != nil {
            return fmt.Errorf("登入失敗: %w", err)
        }
    }

    // 2. 取得發票列表
    invoiceList, err := s.scraper.GetAllInvoices(startDate, endDate)
    if err != nil {
        return fmt.Errorf("取得發票列表失敗: %w", err)
    }

    log.Printf("📋 共取得 %d 筆發票", len(invoiceList))

    // 3. 遍歷每張發票
    for _, inv := range invoiceList {
        // 3.1 檢查是否已存在
        exists, err := s.invoiceRepo.ExistsByNumber(inv.InvoiceNumber)
        if err != nil {
            return err
        }
        if exists {
            log.Printf("⏭️  跳過已存在的發票: %s", inv.InvoiceNumber)
            continue
        }

        // 3.2 取得發票明細
        detail, err := s.scraper.GetInvoiceDetail(inv.Token)
        if err != nil {
            log.Printf("⚠️  取得明細失敗 (%s): %v", inv.InvoiceNumber, err)
            continue
        }

        // 3.3 轉換時間格式（Unix Timestamp -> time.Time）
        invoiceDate := parseUnixTimestamp(inv.InvoiceDate)

        // 3.4 建立 Invoice Entity
        invoice := &domain.Invoice{
            InvoiceNumber: inv.InvoiceNumber,
            InvoiceDate:   invoiceDate,
            SellerName:    inv.SellerName,
            TotalAmount:   decimal.NewFromInt(int64(inv.TotalAmount)),
            RawDetails:    detail, // GORM 自動序列化為 JSONB
            Status:        "PENDING",
        }

        // 3.5 寫入資料庫
        if err := s.invoiceRepo.Create(invoice); err != nil {
            return fmt.Errorf("寫入發票失敗 (%s): %w", inv.InvoiceNumber, err)
        }

        log.Printf("✅ 成功儲存發票: %s (%s - NT$%d)",
            inv.InvoiceNumber, inv.SellerName, inv.TotalAmount)

        // 3.6 避免請求過快（禮貌性延遲）
        time.Sleep(2 * time.Second)
    }

    return nil
}

func parseUnixTimestamp(ts string) time.Time {
    // 財政部 API 回傳的是毫秒級 Unix Timestamp
    msec, _ := strconv.ParseInt(ts, 10, 64)
    return time.Unix(0, msec*int64(time.Millisecond))
}
```

### 6.5 爬蟲執行頻率
* **每日排程:** 每天凌晨 02:00 執行一次完整同步 (避開尖峰時段)
* **手動觸發:** 使用者可透過 API 手動觸發同步 (限制每小時最多 1 次)

### 6.6 錯誤處理機制
* **Cloudflare 挑戰失敗:** 記錄錯誤並通知開發者（可能需要調整等待時間）
* **Session 過期:** 自動重新登入 (最多重試 3 次)
* **JSON Schema 變更:** 使用 `json.Unmarshal` 檢測 API 結構變更，發送警告
* **網路逾時:** 指數退避重試 (Exponential Backoff: 5s, 10s, 30s)
* **圖形驗證碼:** 第一次登入時暫停並要求人工輸入，或使用 OCR 服務

### 6.7 圖形驗證碼處理模組 (`pkg/einvoice/captcha`)

#### 技術選型：CLI vs CGO

3. **重試機制** (`Handler.SolveWithRetry`)

   - 自動調整預處理參數（對比度、閾值）

4. **Fallback 機制** (`Handler.SolveWithFallback`)

## 7. Phase 2.6: Invoice Synchronization Architecture

**Version:** 1.0.0
**Completed:** 2026-02-07

Phase 2.6 完成了發票同步功能的資料庫整合，實現了從爬蟲抓取到資料持久化的完整數據流。

### 7.1 Data Flow Diagram (資料流圖)

```
財政部 API → Scraper (pkg/einvoice)
           → SyncService (usecase layer)
           → InvoiceRepository (repository layer)
           → PostgreSQL (JSONB storage)
```

**流程說明：**

1. **Scraper Layer** (`pkg/einvoice`)
   - 使用 Playwright 攔截財政部 API (`searchCarrierInvoice`)
   - 解析 JSON Response 並轉換為 Go Structs
   - 回傳結構化的發票資料

2. **Usecase Layer** (`internal/usecase/sync_service.go`)
   - 協調爬蟲與資料庫操作
   - 處理重複檢查邏輯（使用 `invoice_number` 查詢）
   - 實作錯誤處理與日誌記錄
   - 累積部分失敗的錯誤並繼續處理

3. **Repository Layer** (`internal/repository/invoice_repository.go`)
   - 實作 GORM 資料庫操作
   - 提供 CRUD 介面 (`Create`, `ExistsByNumber`, `FindByStatus`)
   - 處理 JSONB 資料序列化

4. **PostgreSQL**
   - 儲存結構化欄位 (`invoice_number`, `invoice_date`, `seller_name`, `total_amount`)
   - 使用 JSONB 儲存原始 API Response (`raw_details`)
   - 透過 `invoice_number UNIQUE` 防止重複

### 7.2 Key Design Decisions (關鍵設計決策)

#### 7.2.1 Duplicate Detection (重複偵測)

**策略：** 使用資料庫層面的 `UNIQUE` 約束

```sql
CREATE UNIQUE INDEX idx_invoices_invoice_number
ON invoices(invoice_number);
```

**理由：**
- 確保資料唯一性（Database-enforced integrity）
- 避免應用層競態條件（Race condition）
- 簡化錯誤處理邏輯（Constraint violation = 已存在）

**實作：**
```go
// 1. 查詢前檢查
exists, err := s.invoiceRepo.ExistsByNumber(invoice.InvoiceNumber)
if exists {
    log.Printf("跳過已存在的發票: %s", invoice.InvoiceNumber)
    continue
}

// 2. 寫入時自動防護（UNIQUE constraint）
err := s.invoiceRepo.Create(invoice)
if isDuplicateError(err) {
    // 靜默跳過或記錄警告
}
```

#### 7.2.2 JSONB Storage for Flexibility (JSONB 儲存彈性資料)

**策略：** 分離結構化與非結構化資料

```go
type Invoice struct {
    // 結構化欄位（用於查詢、索引）
    InvoiceNumber string          `gorm:"column:invoice_number;uniqueIndex"`
    InvoiceDate   time.Time       `gorm:"column:invoice_date"`
    SellerName    string          `gorm:"column:seller_name"`
    TotalAmount   decimal.Decimal `gorm:"column:total_amount;type:decimal(19,4)"`

    // 原始資料（用於完整保存、未來分析）
    RawDetails    datatypes.JSON  `gorm:"column:raw_details;type:jsonb"`
}
```

**優點：**
- 保留完整 API Response（未來需求變更時可重新解析）
- 避免過度正規化（不需為每個細節建立資料表）
- PostgreSQL JSONB 支援索引與查詢

#### 7.2.3 Status-based Processing Pipeline (狀態驅動處理流水線)

**狀態流轉：**
```
PENDING → PROCESSED → MATCHED
```

**定義：**
- **PENDING**: 剛從財政部同步，尚未處理
- **PROCESSED**: 已完成正規化（匹配商家規則）
- **MATCHED**: 已成功匹配到商家並建立交易紀錄

**查詢範例：**
```go
// 取得所有待處理的發票
pendingInvoices, err := invoiceRepo.FindByStatus("PENDING")

// 處理後更新狀態
invoice.Status = "PROCESSED"
invoiceRepo.Update(invoice)
```

**優點：**
- 支援批次處理（Worker 定期處理 PENDING 發票）
- 可追蹤處理進度
- 錯誤恢復簡單（重新處理特定狀態的發票）

#### 7.2.4 Context-aware Logging (情境感知日誌)

**策略：** 使用結構化日誌記錄關鍵資訊

```go
log.Printf("[SyncService] 開始同步發票: start=%s end=%s",
    startDate.Format("2006-01-02"),
    endDate.Format("2006-01-02"))

log.Printf("[SyncService] 發票已存在，跳過: invoice_number=%s",
    invoice.InvoiceNumber)

log.Printf("[SyncService] 成功儲存發票: invoice_number=%s seller=%s amount=%s",
    invoice.InvoiceNumber,
    invoice.SellerName,
    invoice.TotalAmount.String())
```

**日誌規範：**
- 前綴標記模組名稱 `[SyncService]`
- 包含關鍵識別資訊 (`invoice_number`)
- 記錄失敗原因（不含敏感資料）
- 使用一致的時間格式 (`2006-01-02`)

### 7.3 Error Handling Strategy (錯誤處理策略)

#### 7.3.1 Session Expiration (Session 過期)

**情境：** 爬蟲的 Session Cookie 失效

**策略：** Fail fast + 要求重新登入

```go
invoices, err := s.scraper.GetInvoices(ctx, startDate, endDate)
if err != nil {
    if isSessionExpiredError(err) {
        return fmt.Errorf("session expired, please re-login: %w", err)
    }
    return fmt.Errorf("failed to fetch invoices: %w", err)
}
```

**理由：**
- Session 重建需要使用者互動（輸入驗證碼）
- 自動重試可能觸發財政部防護機制
- 明確告知使用者需要重新授權

#### 7.3.2 Network Errors (網路錯誤)

**情境：** 暫時性網路問題

**策略：** Fail fast + 記錄錯誤

```go
invoices, err := s.scraper.GetInvoices(ctx, startDate, endDate)
if err != nil {
    log.Printf("[SyncService] 網路錯誤: %v", err)
    return fmt.Errorf("network error: %w", err)
}
```

**理由：**
- 同步操作可由 Worker 定期重試（Cron Job）
- 避免無限重試阻塞執行緒
- 記錄錯誤供管理員診斷

#### 7.3.3 Partial Failures (部分失敗)

**情境：** 100 筆發票中，5 筆儲存失敗

**策略：** Continue processing + 累積錯誤

```go
var errors []error

for _, invoice := range invoices {
    if err := s.invoiceRepo.Create(invoice); err != nil {
        errors = append(errors, fmt.Errorf(
            "failed to save invoice %s: %w",
            invoice.InvoiceNumber, err))
        continue // 繼續處理下一筆
    }
}

if len(errors) > 0 {
    log.Printf("[SyncService] 部分發票儲存失敗: %d/%d",
        len(errors), len(invoices))
    // 回傳第一個錯誤，但記錄所有錯誤
    return errors[0]
}
```

**優點：**
- 最大化成功率（不因單筆失敗而全部失敗）
- 提供完整錯誤資訊供除錯
- 失敗的發票可透過日誌追蹤

#### 7.3.4 Database Constraint Violations (資料庫約束違反)

**情境：** 重複的 `invoice_number`

**策略：** 靜默跳過 + 記錄警告

```go
if err := s.invoiceRepo.Create(invoice); err != nil {
    if strings.Contains(err.Error(), "duplicate key") {
        log.Printf("[SyncService] 發票已存在（略過）: %s",
            invoice.InvoiceNumber)
        continue
    }
    return fmt.Errorf("database error: %w", err)
}
```

**理由：**
- 重複資料不是錯誤（冪等性要求）
- 記錄以供審計追蹤
- 避免不必要的錯誤回報

## 8. API 設計規範 (API Design Guidelines)

### 7.1 Response Format

所有 API 回傳統一使用 JSON 結構：

```json
{
  "code": 200,          // 應用程式狀態碼 (非 HTTP Status)
  "message": "Success", // 人類可讀訊息
  "data": { ... }       // 實際 payload (或是 null)
}

```

### 7.2 Error Handling

* **HTTP 400:** 參數驗證錯誤 (Validation Error)。
* **HTTP 401:** Token 無效或過期。
* **HTTP 500:** 伺服器內部錯誤 (DB 連線失敗等)，**不可**直接將 SQL Error 暴露給前端，需 Log 下來並回傳通用錯誤訊息。

### 7.3 Authentication

* 使用 **JWT (JSON Web Token)** 進行無狀態驗證。
* Header: `Authorization: Bearer <token>`
