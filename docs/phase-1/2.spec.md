## 1. 產品願景與目標 (Vision & Goals)

### 1.1 背景與問題 (Problem Statement)

市面上的記帳軟體通常面臨以下痛點：

1. **手動輸入太累：** 使用者難以持之以恆。
2. **資料太髒：** 雲端發票同步後的名稱雜亂（如：「統一百華01店」），導致報表無法閱讀。
3. **信用卡管理斷層：** 記帳軟體只記錄「消費」，卻忽略了「繳卡費」與「資金流動」的時間差，導致使用者不知道現在實際可用的現金有多少。

### 1.2 產品解決方案 (Solution)

打造一款**「工程師思維」的自動化記帳工具**，核心價值為：

1. **全自動同步：** 透過網頁爬蟲 (Web Scraper) 自動登入財政部電子發票整合服務平台並抓取發票資料。
2. **規則引擎 (Rule Engine)：** 讓使用者自訂規則（Regex/關鍵字），自動清洗雜亂的商家名稱並歸類。
3. **資產生命週期管理：** 自動模擬信用卡扣款邏輯，精準呈現資產現況。

### 1.3 核心指標 (Success Metrics)

* **自動歸類率：** 使用者 90% 的發票應透過規則自動歸類，無需手動介入。
* **使用者操作時間：** 每日僅需 < 1 分鐘確認帳務。

## 2. 使用者畫像 (User Persona)

**Target User: Alex (後端工程師/科技愛好者)**

* **特徵：** 擁有多張信用卡，習慣使用電子支付與載具。
* **痛點：** 雖然有記帳習慣，但對於每個月底要手動核對信用卡帳單感到煩躁。
* **需求：** 想要一個 "Set and Forget" 的系統，設定好規則後，系統自動幫我把帳記好。
* **能力：** 理解什麼是 Regex，喜歡高度客製化的工具。

## 3. 功能需求列表 (Functional Requirements)

我們使用 **MoSCoW 法則** 來排列優先級。

### P0: Must Have (MVP 必須包含)

#### A. 帳戶與資產管理 (Asset Management)

* **F-A1:** 使用者能新增不同類型的帳戶（銀行、現金、信用卡）。
* **F-A2:** 針對信用卡，使用者能設定「結帳日」與「繳款截止日」。
* **F-A3:** 使用者能設定信用卡的「自動扣款帳戶」（連結至某個銀行帳戶）。

#### B. 雲端發票整合 (Invoice Integration)

* **F-B1:** 使用者輸入手機條碼與驗證碼後，系統能透過爬蟲成功登入財政部電子發票整合服務平台並維持 Session。
* **F-B2:** 系統需在背景排程（每日）自動爬取新發票資料。
* **F-B3:** 支援顯示發票明細（品項、單價）。
* **F-B4:** 爬蟲需具備錯誤處理機制（如：Session 過期自動重新登入、頁面結構變更偵測）。

#### C. 智慧歸類規則引擎 (Smart Normalization Engine)

* **F-C1 (規則設定):** 使用者能針對特定關鍵字（如 "Uber"）設定規則，將其自動轉換為標準商家（如 "Uber Eats"）並套用預設分類（如 "飲食"）。
* **F-C2 (規則執行):** 新發票進入時，系統必須依序執行使用者設定的規則。
* **F-C3 (手動修正):** 若發票未命中規則，使用者手動修正後，系統應詢問「是否將此操作存為新規則？」。

#### D. 交易紀錄 (Transaction Ledger)

* **F-D1:** 顯示條列式的交易清單（包含自動同步與手動輸入）。
* **F-D2:** 支援複式簿記邏輯（轉帳功能，資產 A 減少、資產 B 增加）。

---

### P1: Should Have (應有功能，若時程趕可延後)

* **F-E1 (自動繳費模擬):** 系統每日檢查信用卡繳款日，若到期，自動產生一筆「銀行 -> 信用卡」的轉帳紀錄，並發送通知。
* **F-E2 (儀表板):** 首頁顯示「本月支出」、「淨資產」與「預估下月卡費」。

### P2: Could Have (未來擴充)

* **F-F1:** 加密貨幣交易所 API 串接。
* **F-F2:** Line Notify 通知機器人。
* **F-F3:** 預算超支提醒。

## 4. 詳細功能規格與驗收標準 (Feature Specs & Acceptance Criteria)

這裡挑選最核心的 **「規則引擎」** 與 **「發票同步」** 進行詳細定義。

### 4.1 功能模組：發票同步與正規化

**User Story:**
作為使用者，我希望我的 Uber Eats 發票進來後，直接顯示為「Uber Eats - 晚餐」，而不是「優食台灣股份有限公司...」，且自動扣款在我的 Cube 卡上。

**流程說明 (User Flow):**

1. 系統透過爬蟲登入財政部平台，爬取到發票 `AB-12345678`，商家名為 `優食台灣股份有限公司`。
2. 系統將爬取的原始資料儲存至資料庫 (含 HTML/JSON 格式備份)。
3. 系統檢查規則列表。
4. 命中規則：`關鍵字 "優食" -> 商家 "Uber Eats" -> 分類 "晚餐" -> 預設支付 "Cube卡"`。
5. 系統建立交易紀錄，並標記狀態為 `已正規化`。

**驗收標準 (Acceptance Criteria):**

1. **AC-01:** 若無規則命中，該筆交易應顯示原始名稱，分類顯示「未分類」，並以顯眼方式（如紅點）提示使用者。
2. **AC-02:** 使用者建立新規則後，系統應提供選項「是否重新套用此規則至歷史資料？」。
3. **AC-03:** 規則必須支援優先級（Priority），避免短關鍵字誤判（例如 `7-11` 與 `7-11 ATM`）。

### 4.2 功能模組：信用卡自動循環

**User Story:**
作為使用者，我希望系統自動幫我「繳卡費」，這樣我的銀行餘額才是準的，我不用每個月自己去記一筆轉帳。

**流程說明 (User Flow):**

1. 使用者設定 `Cube卡`：結帳日 27 號，繳款日次月 12 號，扣款帳戶 `國泰銀行`。
2. 到了 12 號早上 10 點。
3. 系統檢查 `Cube卡` 當前的負債餘額（例如 -15,000 元）。
4. 系統自動建立一筆轉帳：從 `國泰銀行` 轉 15,000 到 `Cube卡`。
5. `Cube卡` 餘額歸零（或變回正數），`國泰銀行` 餘額減少。

**驗收標準 (Acceptance Criteria):**

1. **AC-04:** 若扣款銀行餘額不足，系統仍應執行記帳（允許負值），但發送「餘額不足警告」給使用者。
2. **AC-05:** 自動產生的交易紀錄，備註欄應標註「系統自動扣繳」。

## 5. 介面草圖 (UI Wireframes Description)

雖然這裡不畫圖，但定義畫面重點：

* **頂部卡片：** 總資產 (Net Worth)。

* **列表式呈現：** 左邊顯示「關鍵字」，右邊顯示箭頭指向「目標商家」。
* **新增規則 Modal：**
* 輸入框：原始名稱關鍵字 (支援 Regex)。
* 選擇框：對應商家。
* 預覽區：輸入測試字串，即時顯示是否命中 (Match/No Match)。

## 6. 非功能需求 (Non-Functional Requirements)

1. **資料隱私：** 雖然是 Side Project，但發票資料敏感，資料庫不得對外開放 port，API 需走 HTTPS。
2. **錯誤恢復：** 若財政部 API 掛掉（常見），系統不能 Crash，要在 Log 紀錄錯誤並於 6 小時後重試。
3. **回應速度：** App 操作（如切換頁面）需在 0.1 秒內反應；手動同步發票容許 10-30 秒等待時間。

## 7. 開發階段 (Phasing)

* **Phase 1 (Core):** 只能跑 Script。完成 API 串接、DB 寫入、基礎規則引擎。目標：你可以自己在資料庫看到發票進來且被歸類。
* **Phase 2 (API):** 建立 Go Backend Server，提供 REST API。
* **Phase 3 (UI):** 開發簡單的前端（App 或 Web）來顯示資料。
